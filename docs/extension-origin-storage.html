Extension-origin audio storage

A hidden extension iframe gives the YouTube watch page access to durable extension-origin IndexedDB. The standalone extension page reaches the same database directly, so playback and library management share one store without involving the service worker.

Mental model

Origin, rather than tab or frame, determines which IndexedDB database code can open. Both src/extension-storage.html and index.html run under the extension origin, so their separate IdbStore instances open the same database and object store.

The YouTube page owns playback state, but the extension origin owns replacement-audio blobs. The hidden frame is the bridge between those two ownership boundaries.

Storage operations

Watch-page operations cross the hidden-frame RPC boundary. Library operations remain entirely within the extension origin.

Watch page

Save or load replacement audio

  1. The relay resolves the frame URL

    content.tsx runs in MAIN world and cannot call chrome.runtime.getURL. The ISOLATED-world entry.relay.ts can, so it resolves the extension URL. content.tsx then uses the generic iframe RPC helper for hidden frame creation and load handling. The relay is not part of later storage RPC calls.

  2. MAIN waits for the frame load

    The RPC helper creates a typed window client from the loaded frame's contentWindow.

  3. Audio crosses by structured clone

    StoredAudio, including track blobs, is sent to ExtensionStorageRpcHandlers, or returned from it.

  4. The frame accesses IndexedDB

    audioStorage.storeAudio uses put, while loadAudio uses get(videoId).

Extension tab

List or delete saved videos

  1. The saved view activates its query

    extension-page.tsx calls audioStorage.listAudio only while the saved-videos view is active.

  2. The page reads the shared store directly

    getAll() produces metadata summaries and computes aggregate blob sizes without transferring the audio to another origin.

  3. Deletion uses the video ID key

    After confirmation, deleteAudio(videoId) removes the record and invalidates the library query.

Stored record

One record represents all replacement tracks for one YouTube video. Existing one-blob records are interpreted as a single-track record when loaded or listed.

Field Role Storage behavior
videoId Object-store key Uniquely selects the replacement for a YouTube video.
name Display name Usually the uploaded file or archive name.
tracks[] Replacement audio Each entry stores a track name and Blob; summary size is the sum of all blob sizes.
videoTitle? Library label Captured from YouTube metadata when audio is selected.
savedAt? Recency New records use Date.now(); listings sort descending, with legacy records last.

Boundaries

The mechanism separates playback access from persistence access and narrows each cross-origin interface.

YouTube MAIN world

Owns the panel, video synchronization, and selected-audio state. It only receives two persistence methods: load and store.

YouTube ISOLATED world

MAIN world cannot access chrome.runtime. The relay can, so it calls chrome.runtime.getURL and mounts the web-accessible extension frame. After this bootstrap, storage RPC goes directly between MAIN and the frame.

Hidden storage frame

Runs with extension-origin storage access. Its RPC handler only accepts messages from its parent window and sends responses to the YouTube origin.

Extension tab

Shares the storage origin with the hidden frame and directly owns library-only operations such as listing and deletion.

IndexedDB

Durably stores blobs under the extension origin. The database is independent of YouTube local storage and is shared across extension pages and tabs.

No service worker hop

The background worker is not part of this path. It remains responsible for privileged tab creation and network fetching, not replacement-audio persistence.

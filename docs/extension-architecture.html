Architecture breakdown

YouTube Audio Replacement

One Manifest V3 extension, two user workflows, and four execution contexts. YouTube remains the playback surface for audio replacement, while a separate extension page acquires audio for the future stem separation workflow.

Mental model

The extension is intentionally split by browser capability. MAIN-world scripts can touch YouTube internals. ISOLATED-world scripts can call Chrome extension APIs. The extension page accepts independent YouTube and local-file sources, but never embeds a visible player.

YouTube is the transport clock for replacement playback. The hidden embed exists only to acquire YouTube source audio; local files never cross an RPC boundary.

Request flows

The same generic runtime relay is used in two frames, while parent-to-iframe calls use a separate transferable-aware RPC.

Flow A

Open the stem generator

  1. User selects Prepare stems

    The MAIN-world panel calls the typed background client with the current video ID.

  2. Frame-local bridge forwards the RPC

    window.postMessage crosses MAIN to ISOLATED without leaking across tabs.

  3. Service worker opens a tab

    The ID is validated, added to index.html?videoId=..., and opened with chrome.tabs.create.

The toolbar action is a shorter variant: the service worker parses the active tab URL and opens the same extension page directly.

Flow B

Download the source audio

  1. Extension page eagerly creates a hidden embed

    Page startup begins loading youtube.com/embed/ and caches the RPC promise. The client verifies the frame and origin before accepting its readiness message.

  2. Parent calls embed handlers

    Window RPC carries typed methods, progress callbacks, errors, and transferable buffers.

  3. Embed resolves and selects the format

    The YouTube player API yields complete audio-only formats. The handler prefers Opus and then chooses the largest available format before downloading it in 5 MB ranges.

  4. Service worker performs CDN fetches

    Only HTTPS Googlevideo hosts pass the allow-list.

  5. Result returns as a page-owned File

    The final ArrayBuffer is transferred to the page and wrapped in a File. The YouTube and local-file modes retain their own selection, and YouTube source audio can be saved directly.

Why there are two RPC layers

Window RPC connects the extension page to its child YouTube iframe and supports callbacks plus zero-copy ArrayBuffer transfer.

Runtime relay RPC connects a MAIN-world script to its same-frame ISOLATED-world bridge, then to the service worker through chrome.runtime.sendMessage.

Current milestone: YouTube acquisition and local-file selection are real. Model configuration, model setup, separation, stem playback, and stem downloads are presentation-only UI pending Demucs integration.

Trust boundaries

Each context exists because Chrome exposes different capabilities to it.

MAIN world

Can use YouTube globals, player methods, DOM events, and same-origin player API calls. It cannot directly use extension APIs.

ISOLATED world

Shares the page frame but has Chrome extension API access. It deliberately contains no product logic.

Extension page

Eagerly creates one hidden YouTube iframe, tracks independent YouTube and local source state, and holds selected Files in memory. It embeds no visible YouTube player.

Service worker

Owns privileged tab creation and cross-origin media fetch. The proxy rejects non-Googlevideo hosts.

YouTube embed

Provides the YouTube origin needed for ytcfg, player API extraction, and format URL resolution. Its MAIN-world handler assembles ranges, while its ISOLATED-world relay alone can reach chrome.runtime.

Persistence

The watch-page product stores panel state in localStorage and replacement blobs in YouTube-origin IndexedDB, keyed by video ID.
